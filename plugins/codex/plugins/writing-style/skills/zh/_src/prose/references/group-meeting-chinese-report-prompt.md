# Prompt: rewrite a Chinese research progress report for group meeting

Use this prompt when a user asks for a Chinese progress report that they can present orally in a lab meeting or group meeting.

```text
你要把下面的技术材料改成“组会可讲”的中文汇报提纲，不是 repo 审计，也不是完整论文式报告。

目标读者：
- 组会听众；
- 他们关心现在有没有进展、哪些方法有效、哪些方法失败、下一步该做什么；
- 他们不会耐心听内部框架、文件路径、branch 细节和英文工程术语。

输出要求：

1. 第一节必须是“当前情况和下一步”，用 bullet point 写。
   必须回答：
   - 现在有没有超过原基线的信号？
   - 如果有，是哪个指标、哪个方法、提升多少？
   - 这是不是官网/线上结果？如果不是，要明确说不是。
   - 每个主要指标当前最大问题是什么？
   - 相比上次多考虑了哪些方法或错误类型？
   - 哪些方法有价值，哪些失败或暂停？
   - 下一步优先级是什么？

2. 后面按汇报页组织，而不是按文件来源组织。
   推荐结构：
   - 0. 30 秒版本
   - 1. 这轮到底有没有新信号？
   - 2. 每个指标现在卡在哪里？
   - 3. 相比上次多考虑了哪些问题和方法？
   - 4. 最值得继续的方向是什么，为什么？
   - 5. 外部代码库/外部方法用了哪些，结果如何？
   - 6. 失败路线还有没有价值？
   - 7. 下一步要做什么？
   - 8. 备份：证据路径

3. 每页都要能直接讲。
   可以加入“可以这样讲：”并给 2-4 句口头表达。
   不要只写长段落。

4. 表格要少而准。
   每个表格最多 4 列，列名要中文。
   表格服务于决策，不要为了显得完整而堆信息。

5. 术语必须换成人能听懂的中文。
   把：
   - guardrail 改成“保护指标 / 不能被误伤的指标”
   - component gate 改成“可疑区域筛选 / 过滤不可靠的小块”
   - hosted metric 改成“官网指标 / 官网评分”
   - dry-run 改成“本地预演”
   - remote FP 改成“远端假阳性”
   - proxy 改成“本地观察指标 / 间接指标”
   - topology cleanup 改成“形态修正 / 连通性修正”
   - label mapping 改成“标签转换”
   - case manifest 改成“病例清单”
   - zip layout 改成“压缩包结构”
   - backbone 改成“主干网络”
   - pipeline 改成“流程”

6. 保留以下事实，不要改数字：
   - 日期、指标名、模型名、数据集名；
   - Dice、HD95、fold、样本数；
   - 谁做了 leaderboard/排行榜提交；
   - 哪些结果是本地结果，哪些是官网结果；
   - 哪些方法失败、哪些只是 audit/smoke、哪些真正有指标。

7. 明确证据边界。
   使用：
   - “本地信号”
   - “还不是官网结果”
   - “还不能直接提交”
   - “需要更多折复现”
   - “需要本地提交包预演”
   避免：
   - “证明”
   - “显著”
   - “已经解决”
   - “可以替代现有方案”
   除非材料明确支持。

8. 外部代码库要按状态分类：
   - 已用于候选并有指标；
   - 只做了代码检查/导入检查，没有指标；
   - 还没真正用起来；
   - 失败但可能有价值。

9. 失败路线不能只写失败。
   每条失败路线都要写：
   - 失败表现；
   - 可能原因；
   - 还有没有价值；
   - 当前决策：继续 / 暂停 / 只作备份 / 需要先排查流程。

10. 最后一节才放证据路径。
    文件路径、branch、commit、命令、日志不要出现在主讲部分，除非用户明确要求。

最终检查：
- 第一页能否直接投屏？
- 30 秒版本能否直接念？
- 听众是否能在 1 分钟内知道下一步做什么？
- 有没有还残留 guardrail、component gate、hosted、dry-run、proxy、remote FP 这种内部词？
- 有没有把“本地结果”说成“官网结果”？
```
